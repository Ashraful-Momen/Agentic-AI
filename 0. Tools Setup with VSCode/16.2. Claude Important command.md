চলো একটা বড় প্রজেক্ট ধরে নিয়ে (ধরি, তুমি একটা **e-commerce ওয়েবসাইট** বানাচ্ছো Node.js + React দিয়ে) — এই তিনটা কমান্ড কীভাবে কাজে লাগবে সেটা উদাহরণ দিয়ে বুঝিয়ে দিচ্ছি।

## 1️⃣ `/init` — প্রজেক্ট শুরুতেই একবার

তুমি যখন প্রজেক্ট ফোল্ডারে গিয়ে প্রথমবার Claude Code চালু করবে, তখনই `/init` চালাবে।

**যা হবে:** Claude পুরো প্রজেক্ট স্ক্যান করে একটা `CLAUDE.md` ফাইল বানাবে, যেখানে এরকম কিছু থাকবে:

```markdown
# CLAUDE.md
- এটা একটা MERN স্ট্যাক e-commerce প্রজেক্ট
- Frontend: /client (React + Tailwind)
- Backend: /server (Express + MongoDB)
- চালানোর কমান্ড: npm run dev (root থেকে)
- টেস্ট: npm test (server ফোল্ডারে)
- কোড স্টাইল: ES modules, async/await ব্যবহার করা হয়
```

**কেন দরকার:** পরে তুমি নতুন সেশনে (নতুন টার্মিনাল খুলে) কাজ শুরু করলেও Claude এই ফাইল পড়ে সাথে সাথেই প্রজেক্ট বুঝে যাবে — বারবার এক্সপ্লেইন করতে হবে না।

---

## 2️⃣ `/memory` — প্রজেক্ট-স্পেসিফিক নিয়ম/পছন্দ জমা রাখতে

ধরো কাজ করতে করতে তুমি বুঝলে Claude বারবার ভুল করছে, যেমন — তুমি চাও সব ফাইলে TypeScript ব্যবহার হোক, কিন্তু Claude মাঝে মাঝে plain JS লিখছে।

তখন `/memory` চালিয়ে ওই ফাইলটা খুলে নিজে হাতে যোগ করে দিতে পারো:

```
- সবসময় TypeScript ব্যবহার করবে, plain .js ফাইল বানাবে না
- API রেসপন্স ফরম্যাট সবসময় { success, data, error } এই স্ট্রাকচারে থাকবে
- ডাটাবেজ কানেকশন স্ট্রিং .env ফাইল থেকে নিতে হবে, হার্ডকোড করা যাবে না
```

**কেন দরকার:** এটা `CLAUDE.md`-এর মতোই, কিন্তু এটা তোমার personal নিয়ম/instructions রাখার জায়গা যা Claude প্রতি সেশনে মনে রাখবে — যেন তোমাকে বারবার একই কথা বলতে না হয়।

---

## 3️⃣ `/compact` — কাজ করতে করতে কথোপকথন বড় হয়ে গেলে

ধরো তুমি ৩-৪ ঘণ্টা ধরে কাজ করছো — অনেকগুলো ফাইল এডিট হয়েছে, অনেক এরর ফিক্স হয়েছে, কথোপকথন অনেক লম্বা হয়ে গেছে। তখন Claude স্লো হয়ে যেতে পারে বা টোকেন লিমিটের কাছাকাছি চলে যেতে পারে।

তখন তুমি লিখবে:
```
/compact
```

**যা হবে:** Claude পুরনো কথোপকথন (যেমন — শুরুর দিকের ছোটখাটো ডিবাগিং, যেগুলো এখন আর দরকার নেই) সংক্ষিপ্ত করে দেবে, কিন্তু গুরুত্বপূর্ণ জিনিস (যেমন — তুমি কী কাজ করছিলে, কোন ফাইলে কী পরিবর্তন হয়েছে) মনে রাখবে।

**উদাহরণ:** ধরো তুমি চাও শুধু সাম্প্রতিক পেমেন্ট গেটওয়ে ইন্টিগ্রেশনের কনটেক্সট রাখতে, বাকি পুরনো আলোচনা (যেমন প্রথমদিকের UI ডিজাইন নিয়ে কথা) বাদ দিতে — তখন লিখতে পারো:
```
/compact পেমেন্ট গেটওয়ে ইন্টিগ্রেশন সংক্রান্ত কথাগুলো রাখো, বাকিটা ছেঁটে ফেলো
```

---

### সংক্ষেপে মনে রাখার সহজ উপায়:

| কমান্ড | কখন ব্যবহার করবে |
|---|---|
| `/init` | প্রজেক্ট শুরুতে **একবার** — Claude-কে প্রজেক্ট চেনানোর জন্য |
| `/memory` | মাঝেমধ্যে — তোমার **নিজের নিয়ম/পছন্দ** যোগ করতে |
| `/compact` | কাজ **অনেক বড়** হয়ে গেলে — কনটেক্সট হালকা করতে |

================================== Important Command for large project ==================================

বড় প্রজেক্টে কাজ করলে সবচেয়ে ভালো উপায় হলো — **প্রজেক্টের মধ্যেই কিছু ফাইলে (Markdown) সব কিছু লিখে রাখা**, যাতে Claude প্রতিবার নতুন সেশনে সেই ফাইলগুলো পড়ে বুঝে যায় কী হয়েছে, কী বাকি। নিচে ধাপে ধাপে একটা প্র্যাক্টিক্যাল ওয়ার্কফ্লো দিচ্ছি।

## ধাপ ১: প্রজেক্টের গোড়ায় কিছু ট্র্যাকিং ফাইল বানাও

তোমার প্রজেক্ট রুটে এই ফাইলগুলো রাখতে পারো:

```
/CLAUDE.md          → প্রজেক্টের overview, স্ট্রাকচার, কনভেনশন (/init দিয়ে বানানো)
/PROGRESS.md         → এখন পর্যন্ত কী কী ফিচার শেষ হয়েছে
/TODO.md             → পরবর্তী ফিচারগুলোর তালিকা, প্ল্যান
```

**PROGRESS.md** এর উদাহরণ:
```markdown
# Progress Log

## ✅ Done
- User authentication (JWT) - backend + frontend
- Product listing page + API
- Cart functionality (backend done, frontend in progress)

## 🚧 In Progress
- Cart UI (client/src/components/Cart.jsx)

## Notes
- Auth middleware: server/middleware/auth.js
- সব API রেসপন্স { success, data, error } ফরম্যাটে
```

## ধাপ ২: প্রতিটা সেশনের শুরুতে Claude-কে পড়াও

নতুন কাজ শুরুর আগে বলবে:
```
CLAUDE.md, PROGRESS.md, আর TODO.md পড়ে নাও, তারপর বলো এখন কী অবস্থায় আছি
```

Claude তখন ফাইল পড়ে context নিয়ে নেবে — তোমাকে আর প্রতিবার explain করতে হবে না।

## ধাপ ৩: নতুন ফিচার শুরুর আগে প্ল্যান করাও (code লেখার আগে)

সরাসরি "এই ফিচার বানাও" না বলে, আগে প্ল্যান চাও:
```
পরের ফিচার হলো "Wishlist"। কোড লেখার আগে backend আর frontend এ কী কী ফাইল/চেঞ্জ লাগবে সেটার একটা প্ল্যান দাও। এখনই কোড লিখো না।
```
এতে করে তুমি প্ল্যানটা রিভিউ করতে পারবে, ভুল দিক আগেই ধরা পড়বে।

## ধাপ ৪: Backend আগে, তারপর Frontend (বা একসাথে ধাপে ধাপে)

```
প্রথমে backend implement করো:
- Wishlist model + schema
- POST /api/wishlist, GET /api/wishlist, DELETE /api/wishlist/:id
টেস্ট করে দেখাও এগুলো কাজ করছে।
```
এটা কাজ শেষ হলে:
```
এখন frontend এ Wishlist button আর page বানাও, এই backend API গুলো use করে
```

## ধাপ ৫: ফিচার শেষ হলে PROGRESS.md আপডেট করাও

```
Wishlist ফিচার শেষ হয়েছে। PROGRESS.md ফাইলটা আপডেট করে দাও — নতুন কী যোগ হলো, কোন ফাইলগুলো পরিবর্তন হলো সেটা লিখে দাও।
```

## ধাপ ৬: `/compact` আর `/memory` কখন ব্যবহার করবে

- কথোপকথন অনেক বড় হয়ে গেলে (একটা ফিচার শেষ, নতুনটা শুরুর আগে) → `/compact`
- কোনো নিয়ম বারবার বলতে হচ্ছে (যেমন "সবসময় TypeScript ব্যবহার করো") → `/memory` দিয়ে একবারই লিখে রাখো

---

### সংক্ষেপে ওয়ার্কফ্লো:

1. `/init` → CLAUDE.md বানাও (একবার)
2. PROGRESS.md + TODO.md ম্যানুয়ালি বানাও
3. প্রতি সেশনে শুরুতে এই ফাইলগুলো পড়াও
4. নতুন ফিচার → আগে প্ল্যান করাও → backend করাও → frontend করাও → টেস্ট করাও
5. শেষে PROGRESS.md আপডেট করাও
6. দরকার হলে `/compact` আর `/memory` ব্যবহার করো

এভাবে করলে প্রজেক্ট যত বড়ই হোক, Claude প্রতিবার সহজেই "কোথায় ছিলাম" সেটা বুঝে নিতে পারবে।
